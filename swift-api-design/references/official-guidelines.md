# Official Swift API Design Guidelines

Source:
- Official page: https://swift.org/documentation/api-design-guidelines/
- Source document: https://github.com/swiftlang/swift-org-website/blob/main/documentation/api-design-guidelines/index.md

This file is a local reference for the official Swift API Design Guidelines. It keeps the official structure, preserves the important examples, and reformats the content into agent-friendly Markdown for offline skill use.

## Introduction

Swift API design is largely about naming and usage patterns that make code feel native to the language and ecosystem.

Core framing:
- Clarity at the point of use is the top priority.
- Clarity matters more than brevity.
- Documentation comments are part of API design, not an afterthought.

## Fundamentals

### Clarity at the point of use

- A declaration is written once but read at call sites many times.
- Evaluate API shape from concrete uses, not only from the declaration.

### Clarity is more important than brevity

- Shorter is not automatically better.
- Add words when they reduce ambiguity.

### Write a documentation comment for every declaration

- If the API is hard to describe simply, it may be the wrong API.
- Begin with a summary.
- Describe what a function does and returns.
- Describe what a subscript accesses.
- Describe what an initializer creates.
- For other declarations, describe what the entity is.

Example summaries:

```swift
/// Returns a "view" of `self` containing the same elements in
/// reverse order.
func reversed() -> ReverseCollection
```

```swift
/// Inserts `newHead` at the beginning of `self`.
mutating func prepend(_ newHead: Int)

/// Returns a `List` containing `head` followed by the elements
/// of `self`.
func prepending(_ head: Element) -> List

/// Removes and returns the first element of `self` if non-empty;
/// returns `nil` otherwise.
mutating func popFirst() -> Element?
```

```swift
/// Accesses the `index`th element.
subscript(index: Int) -> Element { get set }
```

```swift
/// Creates an instance containing `n` repetitions of `x`.
init(count n: Int, repeatedElement x: Element)
```

```swift
/// A collection that supports equally efficient insertion/removal
/// at any position.
struct List {
    /// The element at the beginning of `self`, or `nil` if `self`
    /// is empty.
    var first: Element?
}
```

Longer doc comment structure:

```swift
/// Writes the textual representation of each element of `items`
/// to the standard output.
///
/// The textual representation for each item `x` is generated by
/// the expression `String(x)`.
///
/// - Parameter separator: Text printed between items.
/// - Parameter terminator: Text printed at the end.
/// - Note: To print without a trailing newline, pass `terminator: ""`.
/// - SeeAlso: `CustomDebugStringConvertible`,
///   `CustomStringConvertible`, `debugPrint`.
public func print(
    _ items: Any...,
    separator: String = " ",
    terminator: String = "\n"
)
```

## Naming

### Promote clear usage

#### Include all the words needed to avoid ambiguity

Good:

```swift
extension List {
    public mutating func remove(at position: Index) -> Element
}

employees.remove(at: x)
```

Bad:

```swift
employees.remove(x) // unclear: are we removing x?
```

#### Omit needless words

Bad:

```swift
public mutating func removeElement(_ member: Element) -> Element?

allViews.removeElement(cancelButton)
```

Good:

```swift
public mutating func remove(_ member: Element) -> Element?

allViews.remove(cancelButton)
```

#### Name variables, parameters, and associated types according to roles

Bad:

```swift
var string = "Hello"

protocol ViewController {
    associatedtype ViewType: View
}

class ProductionLine {
    func restock(from widgetFactory: WidgetFactory)
}
```

Good:

```swift
var greeting = "Hello"

protocol ViewController {
    associatedtype ContentView: View
}

class ProductionLine {
    func restock(from supplier: WidgetFactory)
}
```

Tightly-bound associated type naming:

```swift
protocol Sequence {
    associatedtype Iterator: IteratorProtocol
}

protocol IteratorProtocol { }
```

#### Compensate for weak type information

Bad:

```swift
func add(_ observer: NSObject, for keyPath: String)

grid.add(self, for: graphics) // vague
```

Good:

```swift
func addObserver(_ observer: NSObject, forKeyPath path: String)

grid.addObserver(self, forKeyPath: graphics)
```

### Strive for fluent usage

#### Prefer names that form grammatical English phrases

Good:

```swift
x.insert(y, at: z)
x.subviews(havingColor: y)
x.capitalizingNouns()
```

Bad:

```swift
x.insert(y, position: z)
x.subviews(color: y)
x.nounCapitalize()
```

Acceptable longer-tail fluency:

```swift
AudioUnit.instantiate(
    with: description,
    options: [.inProcess],
    completionHandler: stopProgressBar
)
```

#### Begin factory method names with `make`

```swift
x.makeIterator()
```

#### Initializer and factory first arguments should not form a phrase starting with the base name

Good:

```swift
let foreground = Color(red: 32, green: 64, blue: 128)
let newPart = factory.makeWidget(gears: 42, spindles: 14)
let ref = Link(target: destination)
```

Bad:

```swift
let foreground = Color(havingRGBValuesRed: 32, green: 64, andBlue: 128)
let newPart = factory.makeWidget(havingGearCount: 42, andSpindleCount: 14)
let ref = Link(to: destination)
```

Value-preserving conversion:

```swift
let rgbForeground = RGBColor(cmykForeground)
```

#### Name functions and methods according to side effects

Nonmutating names should usually read like noun phrases:

```swift
x.distance(to: y)
i.successor()
```

Side-effecting names should usually read like imperative verb phrases:

```swift
print(x)
x.sort()
x.append(y)
```

#### Name mutating and nonmutating pairs consistently

Verb-based pair examples:

```swift
x.sort()
let z = x.sorted()

x.append(y)
let z2 = x.appending(y)
```

```swift
/// Reverses `self` in-place.
mutating func reverse()

/// Returns a reversed copy of `self`.
func reversed() -> Self
```

```swift
/// Strips all the newlines from `self`.
mutating func stripNewlines()

/// Returns a copy of `self` with all the newlines stripped.
func strippingNewlines() -> String
```

Noun-based pair examples:

```swift
let x = y.union(z)
y.formUnion(z)

let j = c.successor(i)
c.formSuccessor(&i)
```

#### Boolean names should read as assertions

```swift
x.isEmpty
line1.intersects(line2)
```

#### Protocols and type-like names

Identity protocols as nouns:

```swift
protocol Collection { }
```

Capability protocols with `able`, `ible`, or `ing`:

```swift
protocol Equatable { }
protocol ProgressReporting { }
```

Types, properties, variables, and constants should read as nouns.

### Use terminology well

#### Avoid obscure terms when common words work

- Do not choose specialized language unless it preserves important precision.

#### Stick to established meaning

- Do not redefine accepted technical terms.

#### Avoid abbreviations

- Non-standard abbreviations make APIs harder to learn and search.

#### Embrace precedent

Prefer:

```swift
Array
sin(x)
```

Avoid:

```swift
List
verticalPositionOnUnitCircleAtOriginOfEndOfRadiusWithAngle(x)
```

## Conventions

### Document complexity of computed properties that are not O(1)

- Property syntax suggests cheap access, so document non-trivial work explicitly.

### Prefer methods and properties to free functions

Free functions are mainly for cases like:

```swift
min(x, y, z)
print(x)
sin(x)
```

### Follow case conventions

Good acronym handling:

```swift
var utf8Bytes: [UTF8.CodeUnit]
var isRepresentableAsASCII = true
var userSMTPServer: SecureSMTPServer
```

Other acronyms treated as ordinary words:

```swift
var radarDetector: RadarScanner
var enjoysScubaDiving = true
```

### Methods can share a base name when semantics are aligned

Good:

```swift
extension Shape {
    func contains(_ other: Point) -> Bool { ... }
    func contains(_ other: Shape) -> Bool { ... }
    func contains(_ other: LineSegment) -> Bool { ... }
}
```

Also fine across distinct domains:

```swift
extension Collection where Element: Equatable {
    func contains(_ sought: Element) -> Bool { ... }
}
```

Bad: same base name, different meaning

```swift
extension Database {
    /// Rebuilds the database's search index.
    func index() { ... }

    /// Returns the `n`th row in the given table.
    func index(_ n: Int, inTable: TableID) -> TableRow { ... }
}
```

Bad: overload only by return type

```swift
extension Box {
    func value() -> Int? { ... }
    func value() -> String? { ... }
}
```

## Parameters

### Choose parameter names to serve documentation

Good:

```swift
/// Return an `Array` containing the elements of `self`
/// that satisfy `predicate`.
func filter(_ predicate: (Element) -> Bool) -> [Generator.Element]

/// Replace the given `subRange` of elements with `newElements`.
mutating func replaceRange(_ subRange: Range, with newElements: [E])
```

Bad:

```swift
/// Return an `Array` containing the elements of `self`
/// that satisfy `includedInResult`.
func filter(_ includedInResult: (Element) -> Bool) -> [Generator.Element]

/// Replace the range of elements indicated by `r` with
/// the contents of `with`.
mutating func replaceRange(_ r: Range, with: [E])
```

### Take advantage of defaulted parameters

Bad call site:

```swift
let order = lastName.compare(
    royalFamilyName,
    options: [],
    range: nil,
    locale: nil
)
```

Good call site:

```swift
let order = lastName.compare(royalFamilyName)
```

Preferred API:

```swift
extension String {
    public func compare(
        _ other: String,
        options: CompareOptions = [],
        range: Range? = nil,
        locale: Locale? = nil
    ) -> Ordering
}
```

Less desirable method family:

```swift
extension String {
    public func compare(_ other: String) -> Ordering
    public func compare(_ other: String, options: CompareOptions) -> Ordering
    public func compare(_ other: String, options: CompareOptions, range: Range) -> Ordering
    public func compare(_ other: String, options: StringCompareOptions, range: Range, locale: Locale) -> Ordering
}
```

### Prefer defaulted parameters toward the end

- Required arguments usually communicate the stable, essential pattern of use.

### Prefer `#fileID` for production APIs

- Use `#filePath` mainly for tooling, tests, scripts, or workflows where the full path is intentionally useful.
- Use `#file` only when source compatibility requires it.

## Argument Labels

### Omit all labels when arguments cannot be usefully distinguished

```swift
min(number1, number2)
zip(sequence1, sequence2)
```

### In value-preserving conversions, omit the first argument label

```swift
Int64(someUInt32)
```

```swift
extension String {
    init(_ x: BigInt, radix: Int = 10)
}

text += String(veryLargeNumber)
text += String(veryLargeNumber, radix: 16)
```

Narrowing conversions should usually use a descriptive label:

```swift
extension UInt32 {
    init(_ value: Int16)
    init(truncating source: UInt64)
    init(saturating valueToApproximate: UInt64)
}
```

### When the first argument is part of a prepositional phrase, give it a label

```swift
x.removeBoxes(havingLength: 12)
```

Bad:

```swift
a.move(toX: b, y: c)
a.fade(fromRed: b, green: c, blue: d)
```

Good:

```swift
a.moveTo(x: b, y: c)
a.fadeFrom(red: b, green: c, blue: d)
```

### Otherwise, if the first argument forms part of a grammatical phrase, omit its label

Good:

```swift
x.addSubview(y)
view.dismiss(animated: false)
let text = words.split(maxSplits: 12)
let studentsByName = students.sorted(isOrderedBefore: Student.namePrecedes)
```

Bad:

```swift
view.dismiss(false)
words.split(12)
```

### Label all other arguments

- Once the first argument rule is satisfied, the rest should normally remain labeled.

## Special Instructions

### Label tuple members and name closure parameters where they appear in your API

```swift
/// Ensure that we hold uniquely-referenced storage for at least
/// `requestedCapacity` elements.
///
/// If more storage is needed, `allocate` is called with
/// `byteCount` equal to the number of maximally-aligned bytes
/// to allocate.
///
/// - Returns:
///   - reallocated: `true` if a new block of memory was allocated.
///   - capacityChanged: `true` if `capacity` was updated.
mutating func ensureUniqueStorage(
    minimumCapacity requestedCapacity: Int,
    allocate: (_ byteCount: Int) -> UnsafePointer<Void>
) -> (reallocated: Bool, capacityChanged: Bool)
```

### Take extra care with unconstrained polymorphism

Bad overload set:

```swift
struct Array {
    public mutating func append(_ newElement: Element)
    public mutating func append(_ newElements: S) where S.Generator.Element == Element
}
```

Ambiguous use when `Element == Any`:

```swift
var values: [Any] = [1, "a"]
values.append([2, 3, 4]) // nested array or appended contents?
```

Good fix:

```swift
struct Array {
    public mutating func append(_ newElement: Element)
    public mutating func append(contentsOf newElements: S) where S.Generator.Element == Element
}
```
